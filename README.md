# Docker apps for cyberbits.org

Docker container orchestration with OpenTofu. Cloud agnostic: works on any Docker host.

Traefik runs as the central reverse proxy. All applications connect to the `traefik` network and get automatic SSL certificates via Let's Encrypt.

## Requirements

- OpenTofu 1.6+
- Target server with Docker installed
- Deploy user in the `docker` group
- SSH access to target server
- DNS records pointing your domains to the server

## Domain name resolution required

Before running this configuration, you MUST ensure your domain names correctly resolve to your servers.

Let's Encrypt requires valid DNS records to issue SSL certificates:
- Create A records (or AAAA for IPv6) pointing your domain(s) to your server's IP
- Wait for DNS propagation (can take minutes to hours)
- Verify with `dig +short sub.yourdomain.com @1.1.1.1`

Without proper DNS resolution, SSL certificate generation will fail and your sites won't be accessible via HTTPS. Also, this will result in (temporary) Let's Encrypt bans, which are a real PITA when you want to set things up.

## Secrets and state

All application secrets (database passwords, WordPress keys/salts, Nextcloud admin passwords) are auto-generated by OpenTofu on first apply using the `random_password` resource. You never need to set them manually.

Generated credentials are stored in two places:
- `.env` files on the server. For example `/opt/wordpress-apps/{name}/.env`, `/opt/nextcloud-apps/{name}/.env`. These are written with `0600` permissions
- OpenTofu state file (`terraform.tfstate`). Contains all secrets in plaintext

### Protect your state file

The state file is sensitive, treat it like a password store.

For production, use remote state with encryption (e.g., S3 with server-side encryption or OpenTofu Cloud).

### Adding a new app

Just add a new entry to the relevant list in `terraform.tfvars` and run `tofu apply`. New passwords are generated automatically.

## Quick Start

### 1. Install OpenTofu

```bash
brew install opentofu
```

### 2. Configure variables

Copy the example variables file and edit it:

```bash
cp terraform.tfvars.example terraform.tfvars
vim terraform.tfvars
```

### 3. Initialize OpenTofu

```bash
tofu init
```

This downloads the required providers (Docker, Random, Local).

### 4. Deploy your apps

```bash
tofu plan
tofu apply
```

## Application configuration

### Nginx static sites

Add to your `terraform.tfvars`:

```hcl
nginx_apps = [
  {
    name    = "mysite"
    domain  = "example.com"
    title   = "My Site"
    message = "Welcome to my site"
    www_redirect = false
    # custom_nginx_conf = false  # Set to true to use custom nginx.conf
    # memory_limit = 64          # Container memory limit in MiB
    # cpu_shares   = 512         # Container CPU shares
  }
]
```

**Adding website content**

After deployment, website content is stored in `/opt/nginx-apps/{name}/data/` on the server.
Simply upload your HTML/CSS/JS files to this directory.

Example:
```bash
# Deploy the site
tofu apply

# Upload your website
scp -r ./my-website/* user@server:/opt/nginx-apps/mysite/data/
```

**Custom Nginx configuration**

To use a custom nginx.conf:

1. Set `custom_nginx_conf = true` in your site config
2. Create `/opt/nginx-apps/{name}/nginx.conf` on the server
3. Re-run `tofu apply`

Example `nginx.conf`:
```nginx
server {
  listen 80;
  server_name _;
  root /var/www/html;
  index index.html index.htm;
  location / {
    try_files $uri $uri/ =404;
  }
}
```

### WordPress apps

Add to your `terraform.tfvars`:

```hcl
wordpress_apps = [
  {
    name         = "blog"
    domain       = "blog.example.com"
    db_name      = "blog_wp"
    db_user      = "blog_wp"
    table_prefix = "wp_"
    www_redirect = false
    # Optional PHP settings:
    php_memory_limit = "512M"
    php_upload_max_filesize = "64M"
    php_post_max_size = "64M"
    php_max_execution_time = 300
    # Optional Redis settings:
    redis_maxmemory = "128mb"
    redis_maxmemory_policy = "allkeys-lru"
  }
]
```

Optional settings (with defaults):

| Setting | Default | Description |
|---------|---------|-------------|
| `php_memory_limit` | 512M | PHP memory per request |
| `php_upload_max_filesize` | 64M | Maximum upload file size |
| `php_post_max_size` | 64M | Maximum POST data size |
| `php_max_execution_time` | 300 | Script timeout in seconds |
| `redis_maxmemory` | 128mb | Redis cache size |
| `redis_maxmemory_policy` | allkeys-lru | Redis eviction policy |
| `app_memory_limit` | 512 | PHP-FPM container memory limit (MiB) |
| `app_cpu_shares` | 1024 | PHP-FPM container CPU shares |
| `nginx_memory_limit` | 128 | Nginx sidecar memory limit (MiB) |
| `nginx_cpu_shares` | 256 | Nginx sidecar CPU shares |
| `db_memory_limit` | 512 | MariaDB container memory limit (MiB) |
| `db_cpu_shares` | 512 | MariaDB container CPU shares |
| `redis_memory_limit` | 256 | Redis container memory limit (MiB) |
| `redis_cpu_shares` | 256 | Redis container CPU shares |
| `wpcli_memory_limit` | 128 | WP-CLI sidecar memory limit (MiB) |
| `wpcli_cpu_shares` | 256 | WP-CLI sidecar CPU shares |

**Accessing credentials**

Database credentials are stored in `/opt/wordpress-apps/{name}/.env` on the server.

### Nextcloud apps

Add to your `terraform.tfvars`:

```hcl
nextcloud_apps = [
  {
    name       = "cloud"
    domain     = "cloud.example.com"
    admin_user = "henkbatelaan"
    # Optional settings:
    php_memory_limit = "1024M"
    max_upload_size = "10G"
    redis_maxmemory = "256mb"
  }
]
```

Optional settings (with defaults):

| Setting | Default | Description |
|---------|---------|-------------|
| `php_memory_limit` | 1024M | PHP memory for Nextcloud |
| `max_upload_size` | 2G | Maximum file upload size |
| `redis_maxmemory` | 256mb | Redis cache size |
| `app_memory_limit` | 1024 | Nextcloud container memory limit (MiB) |
| `app_cpu_shares` | 1024 | Nextcloud container CPU shares |
| `cron_memory_limit` | 256 | Cron container memory limit (MiB) |
| `cron_cpu_shares` | 256 | Cron container CPU shares |
| `db_memory_limit` | 512 | PostgreSQL container memory limit (MiB) |
| `db_cpu_shares` | 512 | PostgreSQL container CPU shares |
| `redis_memory_limit` | 512 | Redis container memory limit (MiB) |
| `redis_cpu_shares` | 256 | Redis container CPU shares |

**Accessing admin credentials**

Admin credentials are stored in `/opt/nextcloud-apps/{name}/.env` on the server:

```bash
cat /opt/nextcloud-apps/cloud/.env
```

Also retrievable via `tofu output -json nextcloud_credentials | jq` on your management workstation.

## Destroying resources

**WARNING** This will delete all containers, volumes, and data!

```bash
# Destroy everything
tofu destroy

# Destroy all instances of a specific module type
tofu destroy -target=module.nginx_apps

# Destroy a single instance
tofu destroy -target='module.nginx_apps["firstnx"]'
```

To keep data volumes but remove containers (WordPress example, Nextcloud works the same way):

```bash
# Remove volumes from tofu state so destroy won't touch them
tofu state rm 'module.wordpress_apps["firstwp"].docker_volume.db_data'
tofu state rm 'module.wordpress_apps["firstwp"].docker_volume.wordpress_data'
tofu state rm 'module.wordpress_apps["firstwp"].docker_volume.redis_data'

# Then destroy the rest of that instance
tofu destroy -target='module.wordpress_apps["firstwp"]'
```

> Nginx sites use bind mounts — their content lives directly in `/opt/nginx-apps/{name}/data/` on the host and is never managed as a Docker volume.

## Remote Docker host

This configuration is designed to deploy to a remote server over SSH.
Set `docker_host` in your `terraform.tfvars`:

```hcl
docker_host = "ssh://deployacc@host.domain.org"
```

Never use TCP (`tcp://host:2375`), it exposes the Docker daemon without TLS.

## Maintenance

### Update images

Use `-replace` to force a container to be destroyed and recreated with the latest image.
This is safe — only the container is replaced, Docker volumes (and your data) are preserved.

```bash
# Traefik gateway
tofu apply -replace=module.traefik.docker_container.traefik

# Nginx static site (replace 'firstnx' with your site name)
tofu apply -replace=module.nginx_apps[\"firstnx\"].docker_container.nginx

# WordPress site — replace the PHP-FPM and Nginx containers
tofu apply -replace=module.wordpress_apps[\"firstwp\"].docker_container.wordpress \
           -replace=module.wordpress_apps[\"firstwp\"].docker_container.nginx

# Nextcloud instance
tofu apply -replace=module.nextcloud_apps[\"firstnc\"].docker_container.app
```

Do not use `tofu destroy` to update images. `tofu destroy -target=module.wordpress_apps` removes the Docker volumes too, permanently deleting all database data and uploaded files.

### Backup data

```bash
# Backup Nginx (bind mount — files live directly on the host)
sudo tar -czf nginx-backup.tar.gz /opt/nginx-apps/

# Backup WordPress
sudo tar -czf wordpress-backup.tar.gz /opt/wordpress-apps/
docker run --rm -v wp-firstwp-wordpress-data:/data -v $(pwd):/backup alpine tar -czf /backup/wp-data.tar.gz -C /data .

# Backup Nextcloud
sudo tar -czf nextcloud-backup.tar.gz /opt/nextcloud-apps/
docker run --rm -v nc-firstnc-nextcloud-data:/data -v $(pwd):/backup alpine tar -czf /backup/nc-data.tar.gz -C /data .
```
